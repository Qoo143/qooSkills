# 接收代碼審查

## 概述

代碼審查需要技術評估，而非情緒表演。

**核心原則：** 實施前驗證。假設前詢問。技術正確性優於社交舒適度。

## 回應模式

```
當收到代碼審查反饋時：

1. 閱讀：完整反饋，不做反應
2. 理解：用自己的話重述需求（或詢問）
3. 驗證：對照代碼庫現實檢查
4. 評估：對這個代碼庫技術上合理嗎？
5. 回應：技術性確認或有理由的反駁
6. 實施：一次一項，每項都測試
```

## 禁止的回應

**絕對不要：**
- 「你說得太對了！」（表演性同意）
- 「好觀點！」 / 「很好的反饋！」
- 「讓我現在就實施」（驗證之前）

**應該：**
- 重述技術需求
- 問澄清問題
- 如果錯誤，用技術推理反駁
- 直接開始工作（行動 > 言語）

## 處理不清楚的反饋

```
如果任何項目不清楚：
  停止 - 還不要實施任何東西
  詢問不清楚項目的澄清

為什麼：項目可能相關。部分理解 = 錯誤實施。
```

**示例：**
```
審查者：「修復 1-6」
你理解 1,2,3,6。不清楚 4,5。

❌ 錯誤：現在實施 1,2,3,6，之後再問 4,5
✅ 正確：「我理解項目 1,2,3,6。需要在繼續前澄清 4 和 5。」
```

## YAGNI 檢查「專業」功能

```
如果審查者建議「正確實現」：
  在代碼庫中搜索實際使用情況

  如果未使用：「這個端點沒有被調用。移除它（YAGNI）？」
  如果使用了：那就正確實現
```

## 何時反駁

反駁當：
- 建議會破壞現有功能
- 審查者缺乏完整上下文
- 違反 YAGNI（未使用的功能）
- 對這個技術棧技術上不正確
- 存在遺留/兼容性原因

**如何反駁：**
- 用技術推理，不要防禦性
- 問具體問題
- 引用工作的測試/代碼

## 確認正確的反饋

當反饋確實正確時：
```
✅ 「已修復。[簡短描述改了什麼]」
✅ 「發現得好 - [具體問題]。在 [位置] 修復了。」
✅ [直接修復並在代碼中展示]

❌ 「你說得太對了！」
❌ 「好觀點！」
❌ 「謝謝捕獲到這個！」
❌ 任何感謝表達
```

**為什麼不感謝：** 行動說話。直接修復它。代碼本身展示你聽到了反饋。

## 優雅地糾正你的反駁

如果你反駁了但錯了：
```
✅ 「你是對的 - 我檢查了 [X] 它確實 [Y]。現在實施。」
✅ 「驗證了這個你是正確的。我最初的理解錯誤因為 [原因]。修復中。」

❌ 長篇道歉
❌ 為什麼反駁辯護
❌ 過度解釋
```

陳述事實糾正然後繼續。
