# 構思與設計

## 概述

通過自然的協作對話，將想法轉化為完整的設計和規格。

**核心原則：** 先理解項目現狀，一次問一個問題來細化想法，然後分段呈現設計。

## 流程

### 理解想法
- 先查看當前項目狀態（文件、文檔、最近的變更）
- **一次只問一個問題** 來細化想法
- 盡可能使用選擇題，開放式問題也可以
- 專注於理解：目的、約束、成功標準

### 探索方案
- 提出 2-3 種不同的方案及其權衡
- 以對話方式呈現選項，附上你的建議和理由
- 先說明推薦方案並解釋原因

### 呈現設計
- 當你認為已經理解要構建什麼時，呈現設計
- 分成 200-300 字的小節
- **每節之後詢問是否正確**
- 涵蓋：架構、組件、數據流、錯誤處理、測試
- 如有不清楚之處，準備好回頭澄清

## 設計完成後

**文檔化：**
- 將驗證過的設計寫入 `docs/plans/YYYY-MM-DD-<主題>-design.md`

**繼續實施（如果要繼續）：**
- 問：「準備好開始實施了嗎？」
- 參閱 `writing-plans` 技能創建詳細實施計劃

## 關鍵原則

| 原則 | 說明 |
|------|------|
| 一次一個問題 | 不要用多個問題壓倒用戶 |
| 優先選擇題 | 比開放式問題更容易回答 |
| 嚴格 YAGNI | 從所有設計中移除不必要的功能 |
| 探索替代方案 | 在確定前總是提出 2-3 種方案 |
| 漸進式驗證 | 分段呈現設計，驗證每一段 |
| 保持靈活 | 需要時回頭澄清 |
